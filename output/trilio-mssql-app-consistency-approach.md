# Application-Consistent Microsoft SQL Server Protection on OpenShift Virtualization

### Trilio's approach — validated in the lab

**RPO target: 5 minutes · RTO: data-transfer time + seconds of log replay · Every claim below is lab-verified, not theoretical.**

---

## Executive summary

Trilio for Kubernetes takes **application-consistent** backups of Microsoft SQL
Server running inside a Windows VM on OpenShift Virtualization — and recovers
them to a **point in time**, not just to a crash-consistent snapshot. It does
this using only the standard components already present on an OpenShift Virt
cluster:

- **No external backup server.** The entire workflow runs through the OpenShift
  API. There is no Windows-based orchestrator VM to license, patch, or protect.
- **No vendor agent inside the guest.** Trilio coordinates through the **QEMU
  Guest Agent (QGA)** that Red Hat already ships for OpenShift Virt — which in
  turn drives the native **Windows Volume Shadow Copy Service (VSS)** and the
  **SQL Server VSS Writer**. Nothing proprietary is installed in the VM.
- **Proven end to end.** In lab testing, a backup froze and thawed SQL Server
  cleanly, the database came back **online with zero committed transactions
  lost**, and a subsequent log-chain replay recovered work committed *after*
  the snapshot was taken — demonstrating true point-in-time recovery.

---

## The problem: a VM snapshot is not a database backup

A storage-level or crash-consistent VM snapshot captures the disk as if the
power had been pulled. For a running SQL Server that is dangerous:

- In-flight transactions and buffered writes may be **mid-flight** when the
  snapshot fires.
- On restore, SQL Server must run **crash recovery** — and may roll back
  committed work or, in the worst case, come up **suspect / in repair mode**.
- There is **no way to recover to a chosen moment in time** — you get exactly
  the instant the snapshot was taken, nothing finer.

Protecting SQL Server correctly requires the database engine to be **quiesced
in a known-good state** at the moment the snapshot is taken. That is what VSS
and the SQL Server VSS Writer exist to do — and what Trilio orchestrates.

## How it works: the quiesce handshake

When Trilio takes a backup of the VM, it drives this sequence automatically:

```
Trilio Backup  ──▶  pre-backup lifecycle hook:
                    COPY_ONLY anchor backup written and
                    verified inside the VM — automatic
                        │
                        ▼
                    CSI snapshot request
                        │
                        ▼
                 QGA guest-fsfreeze-freeze
                        │
                        ▼
              Windows VSS framework engages
                        │
                        ▼
        SQL Server VSS Writer quiesces the database
        (flushes buffers, holds writes — Event 3197)
                        │
                        ▼
                 CSI snapshot is taken
                        │
                        ▼
                 QGA guest-fsfreeze-thaw
                        │
                        ▼
        SQL Server VSS Writer resumes the database
        (Events 3198 "thaw" + 18264 "database backed up")
```

The result is a snapshot in which **SQL Server is transactionally consistent**.
Every step from "CSI snapshot request" through "resume" is orchestrated by
Trilio with **no human action and no in-guest software beyond stock Windows +
QGA**.

**The proof is written into the Windows event log by SQL Server itself** — the
3197 / 3198 / 18264 signature is generated by the SQL Server VSS Writer, not by
Trilio, so it is independent evidence that the database genuinely quiesced.

## Point-in-time recovery: past the snapshot moment

An app-consistent snapshot recovers the database to the instant it was taken.
To reach a **5-minute RPO**, Trilio layers the standard SQL Server log-backup
model on top:

- The database runs in **Full recovery model**.
- **Transaction-log backups** (`.trn`) are shipped to S3 on a short cadence
  (e.g. every 5 minutes), giving a continuous, replayable log chain. This is
  the database team's **standard SQL Server maintenance practice** (a SQL
  Server Agent job running `BACKUP LOG ... TO URL='s3://...'`) — it keeps
  working exactly as DBAs run it today, unchanged by Trilio.
- At every Trilio backup, a **pre-backup lifecycle hook automatically takes
  and verifies a COPY_ONLY anchor backup inside the VM** — anchoring the log
  chain to the snapshot without touching the DBA's schedule or breaking their
  backup chain (COPY_ONLY by design does not interrupt it).
- At restore, the app-consistent copy provides the **base**, and the log chain
  is **replayed forward** to the chosen recovery point.

This means committed work that happened **after** the last full snapshot is
still recoverable — the hallmark of point-in-time recovery, and the difference
between "we got last night's backup" and "we lost five minutes."

## What we validated in the lab

A complete backup → cross-namespace restore → point-in-time recovery cycle was
run end to end. Results:

| What was tested | Result |
|---|---|
| **App-consistent quiesce** | SQL Server VSS Writer froze and thawed cleanly — event signature **7× freeze (3197), 7× thaw (3198), 7× "database backed up" (18264)** across the databases. All 12 VSS writers reported **Stable / no error**. |
| **Clean recovery** | Restored VM came up with SQL Server services running and the database **ONLINE** — no recovery/repair mode, **no lost committed transactions**. |
| **Point-in-time, not crash-consistent** | Final row count = **base rows + rows committed *after* the snapshot**, with **original timestamps preserved** — proving the post-snapshot rows arrived via **log replay**, not a snapshot rewind. |
| **Log-chain integrity** | The transaction-log backup's LSN range provably covered the base backup's last LSN, so the replay condition was satisfied and no gap existed in the chain. |
| **Automated in-guest anchor** | On every backup, a Trilio pre-backup lifecycle hook took a COPY_ONLY anchor backup inside the VM and verified it (`RESTORE HEADERONLY`) — **zero operator action**, confirmed by the anchor's own event-log entry and LSN header seconds before the quiesce. |
| **No external infrastructure** | Entire cycle ran through the OpenShift API and the S3 target. No Windows orchestrator VM, no in-guest vendor agent. |

**Representative timings (single VM, ~20 GiB):** app-consistent backup ≈ 7m30s;
cross-namespace restore ≈ 9m10s; in-guest database + log replay ≈ **under 1
second combined**. Backup and restore are **data-transfer-bound** — the SQL
recovery itself is effectively instant.

*(One cosmetic VSS event, 8194, appears on workgroup-joined VMs — a benign
access-control note on an optional callback interface. It does not affect the
SQL Server VSS Writer or the integrity of the backup. Called out here for
completeness because it shows up in the event log.)*

## Who runs what

The design deliberately splits responsibility along the existing operational
boundary — each team keeps the tools it already owns:

| Responsibility | Owner | How |
|---|---|---|
| **Transaction-log cadence** (drives the 5-min RPO) | **Database team** | Their existing SQL Server maintenance schedule (SQL Server Agent) ships `BACKUP LOG TO URL='s3://...'` — standard DBA practice, unchanged. |
| **App-consistent VM backup + chain anchor** | **Trilio (automatic)** | Quiesce handshake plus the pre-backup hook's COPY_ONLY anchor — no schedule, no scripts, no in-guest agent. |
| **Point-in-time restore** | **Trilio-orchestrated** | VM recovery from the app-consistent copy, then log-chain replay from S3 to the chosen recovery point. |

DBAs keep doing log backups exactly as they do today; Trilio makes the
snapshot application-consistent, anchors the chain automatically, and drives
the point-in-time recovery. Both halves were exercised in the lab.

---

## Where this goes next (near-term)

The **backup side is fully automated today** — the quiesce handshake and the
in-guest anchor both run as part of the Trilio backup with zero operator
interaction, as validated above. The near-term engineering track applies the
same lifecycle-hook automation to the **restore-time recovery steps** (the
base restore and log-chain replay), closing the gap between "proven
primitive" and "one-click product experience." Also on the roadmap: surgical
file-level recovery of individual database files directly from a backup.
