# Sub-track 1 POC — TVK virt-launcher hook drives in-guest MSSQL via QGA (2026-07-04)

**Result: PASS.** A TVK `Hook` CR attached to the VM BackupPlan automatically takes a
verified `COPY_ONLY` anchor backup of `demo_db` inside the Windows guest on every
Trilio backup — Exp-4's manual Path A anchor step, fully automated. Validation
backup #7 (`mssql-vss-backup-rlkmn`) completed Available with both hooks green and
a clean 4-second VSS freeze signature.

## What was built

- `manifests/hook-mssql-anchor.yaml` — Hook CR.
  - **pre** (before freeze): `virsh qemu-agent-command` → QGA `guest-exec` →
    `SQLCMD.EXE` as `NT AUTHORITY\SYSTEM` → `BACKUP DATABASE demo_db ... WITH
    COPY_ONLY, INIT, FORMAT` to `D:\SQLBackup\tvk-prehook-anchor.bak`, then
    `RESTORE HEADERONLY` to verify and log the LSN chain.
  - **post** (while frozen): freeze-safe ack only — `guest-fsfreeze-status` +
    `guest-ping` (whitelisted during freeze), returns in ~1s so TVK thaws
    immediately.
- `manifests/backupplan.yaml` — `hookConfig` targeting the launcher pod via
  `podSelector` label `vm.kubevirt.io/name: <vm>` + `containerRegex: ^compute$`.

## Findings (each cost one diagnostic backup, #3–#6)

1. **TVK hook ordering: pre-hook → fsfreeze → CSI snapshot → post-hook → thaw,
   and the thaw waits for the post-hook.** The post-hook therefore runs with the
   guest still frozen, where QGA disables `guest-exec` ("Command guest-exec has
   been disabled: the command is not allowed"). A guest-exec post-hook can never
   succeed — it deadlocks against the thaw it is blocking. Proven by trace:
   180/180 submit rejections across the post-hook's full lifetime (backup #6,
   `hook-poc-backup6-posthook-frozen-trace-excerpt.txt`).
2. **A slow post-hook degrades VSS quality.** Holding the freeze past ~60 s trips
   SQL Server's VSS writer freeze timeout: backups #4–#6 show thaw (`3198`) at
   exactly +60 s and are missing the 7×`18264` completion events (see
   `hook-poc-vss-events-backups3-7-20260704.txt`). The CSI snapshot is still cut
   inside the frozen window, and `demo_db` stayed ONLINE throughout, but the
   clean app-consistent signature requires the post-hook to return in seconds.
3. **TVK pins the Hook's `resourceVersion`/`uid` into
   `BackupPlan.spec.hookConfig` at admission.** Editing a Hook CR does NOT
   propagate to subsequent backups — backup #4 silently re-ran the old hook.
   After any hook edit, re-pin the RV in the BackupPlan (patch command in the
   manifest header).

## Design implication

Guest-exec work (anchors, log backups, app quiesce beyond VSS) belongs in
**pre-hooks only**. Post-snapshot in-guest SQL work cannot ride the backup-side
post-hook; it stays restore-side (Exp-4 pattern) or cluster-side.
